# 001 — DAG vs Serial Execution Mode

> **实验日期**: 2026-06-20
> **状态**: 完成
> **结论**: 假设不成立 — DAG 未优于 Serial，根因是 Planner 平均仅生成 1.2 步

---

## 1. Background

DocMind 的 PER Agent 采用 **DAG（有向无环图）** 执行架构：

1. Planner 将任务分解为多步，标注依赖关系和并行组
2. Executor 按依赖拓扑调度，同一并行组的步并发执行（`asyncio.gather`）
3. Reflector 评估结果

理论优势：对于跨文档分析、多源综合等任务，独立步骤可同时执行，降低延迟。

**假设**: DAG 并行执行在低依赖任务上显著优于串行执行（ReAct 风格）。

## 4. Root Cause Analysis

### 4.1 关键证据: Avg Steps = 1.2

DAG 理论上应该比别人快，但前提是 Planner 生成了多步可并行执行的计划。实际上，**30 题的平均步骤数只有 1.2**。

分布：
- 1 步: 25/30 (83%)
- 2 步: 4/30 (13%)
- 3 步: 1/30 (3%)

当 `step_count ≈ 1` 时，DAG 和 Serial 本质上是同一件事。DAG 的依赖解析和分组调度变成纯开销。

### 4.2 Planner 行为分析

Planner 倾向于把复杂任务压缩成单步。例如：

- **理论 4 步 DAG**: "搜索公司A → 搜索公司B → 搜索公司C → 对比分析"
- **Planner 实际输出 1 步**: "分析这三家公司的差异"

结果：LLM 在一个 prompt 内自行完成所有子任务，Executor 没有参与多步调度。

### 4.3 为什么 Serial 反而更好？

并非 Serial 更好，而是 **LLM 响应中的随机波动**（同一个问题的两次回答不会完全一样）在覆盖率和耗时上造成了约 3-4% 的差异。当 `step_count ≈ 1` 时，两种模式测量的本质是同一个系统，差异来自噪声。

---

## 5. Key Insight

> **当前系统的性能瓶颈不是执行架构，而是任务分解粒度。**

DAG 的并行能力从来没有被真正激活过。Planner 的默认行为——将任务压缩成最少步骤——使得无论选择 DAG 还是 Serial，最终执行路径几乎一样。

这引出了一个更深的研究问题：

### Agent 的最优规划粒度是什么？

| 粒度 | 步骤数 | 预期问题 |
|------|:------:|---------|
| 粗 (coarse) | ~1 | LLM 自行完成所有推理，无并行、不可观察 |
| 适中 (normal) | ~3-4 | 适度拆分，部分并行可得，协调成本低 |
| 细 (fine) | ~8-12 | 最大并行潜力，但上下文被切碎，token 增加 |

关键假设：**成功率 vs 步骤数存在倒 U 曲线关系**。步骤过少 = LLM 独自解决（Agent 无附加价值），步骤过多 = 上下文碎片化，协调成本超过并行收益。中间存在最优区间。

---

## 6. Next Hypothesis

**假设**: Planner 拆解粒度存在最优区间 — coarse（~1 步）和 fine（~12 步）各有利弊，normal（~3-4 步）在成功率和延迟之间取得最佳平衡。

**实验设计**: 引入 `planning_mode` 参数（coarse / normal / fine），控制 Planner 的最大步骤数和拆分倾向，在相同 30 题上对比。
